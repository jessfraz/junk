Getting & Retaining Contributors

Create

help-wanted

labels.

Be aware of any

"internal"

decision making dynamics.

Define

contributions

outside of code.

Value

positive reinforcement

to help people become

repeat contributors.

Be respectful of

contributors time.

Clearly define processes

for people officially becoming

contributors

and

maintainers.

Scaling Maintainers

Create

explicit guidelines

for acceptable patches & release cutoffs.

Clearly define processes

for people officially becoming

contributors

and

maintainers.

Encourage

frequent contributors

to take on more

responsibility.

Distribute control.

Start with the items that are the lowest "cost",

aka can break short-term and no one goes crazy.

Governance

Ability to

change those in power

should not lie with

those in power.

Dealing with Vulnerabilities

If your project is being used, it will have

vulnerabilities discovered,

the key is how you deal with them.

Define a process.

Make the process public.

Handling everyone's needs.

  • Users want software without bugs.
  • Users want to upgrade without breaking the world.
  • Users want to know when to upgrade.
  • Researchers/Bug reporters want updates on what is happening.
  • Project wants to continue to instill trust.

Do not compromise

on disclosure dates

so someone can give a talk at blackhat.

Clearly define the

maximum time frame

for fixing a bug.

Make sure

the process accounts for

when things go wrong.

(ex. vendors breaking embargo)
